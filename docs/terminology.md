# 8주 동안 익힌 용어 정리

PCFILTER QA 인턴십을 하면서 처음에는 낯설었지만, 테스트를 반복하며 의미를 익힌 용어들을 정리했습니다.

이 문서는 사전식 정의보다 “QA 업무에서 어떻게 이해했는가”에 가깝습니다. 내부 정책명, 고객사명, 실제 서버 정보는 제외했습니다.

## 제품과 보안 기능

| 용어 | QA 관점에서 이해한 의미 |
|---|---|
| PCFILTER | 기업 PC에서 개인정보보호와 정보유출방지 기능을 제공하는 보안 제품 |
| 에이전트 | 테스트 Mac에 설치되어 검사, 차단, 정책 반영, 결과 확인을 수행하는 클라이언트 앱 |
| 관리자 페이지 | 정책을 설정하고 로그와 리포트를 확인하는 관리 화면 |
| 개인정보보호 | 파일 안의 개인정보를 검사하고 결과와 보호조치를 확인하는 기능군 |
| DLP | USB, 출력물, 파일 첨부, 공유폴더, 소프트웨어 실행 같은 유출 경로를 제어하는 영역 |
| 보호조치 | 검출된 파일에 대해 암호화, 삭제, 격리 등 후속 처리를 수행하는 기능 |
| 파일 보유 현황 | 검출되거나 보호조치된 파일 상태를 사용자가 확인하는 화면 |

## 테스트와 QA 기록

| 용어 | QA 관점에서 이해한 의미 |
|---|---|
| 재현 | 같은 조건에서 같은 문제가 다시 발생하는지 확인하는 과정 |
| 기대 결과 | 요구사항이나 정책상 나와야 하는 정상 동작 |
| 실제 결과 | 테스트 중 실제로 확인한 화면, 로그, 리포트 결과 |
| 증거 | 개발자나 리뷰어가 다시 확인할 수 있도록 남기는 화면, 로그, 시간, 조건 기록 |
| 회귀 테스트 | 수정된 기능이 기존 흐름을 깨뜨리지 않았는지 다시 확인하는 테스트 |
| 테스트 시나리오 | 사전 조건, 수행 단계, 기대 결과, 확인 기준을 묶은 테스트 흐름 |
| 템플릿 가드 | 자동화가 사람이 작성한 문서를 덮어쓰지 않도록 막는 보호 로직 |

## 정책과 기록

| 용어 | QA 관점에서 이해한 의미 |
|---|---|
| 정책 | 관리자 화면에서 설정하고 에이전트 동작에 영향을 주는 조건 |
| 정책 갱신 | 관리자 설정 변경을 에이전트가 반영하도록 동기화하는 과정 |
| 개별 정책 | 특정 테스트 단말이나 대상에 맞춰 별도로 확인하는 정책 흐름 |
| 로그 | 어떤 동작이 언제 어떤 결과로 처리되었는지 남는 기록 |
| 리포트 | 로그나 검사 결과를 관리자가 확인하기 쉬운 형태로 정리한 결과물 |
| 라이선스 설정 | 기능 노출이나 사용 가능 여부에 영향을 줄 수 있는 상위 설정 |

## 협업 도구와 개발 흐름

| 용어 | QA 관점에서 이해한 의미 |
|---|---|
| M-EP | 팀 온보딩, 가이드, 업무 DB, 제품 문서를 모아둔 Notion 작업 공간 |
| Notion DB | 테스트 요청, 업무 문서, 가이드 등을 구조화해서 찾고 연결하는 공간 |
| Jira 티켓 | 변경 이유, 재현 조건, 기대 결과를 읽고 테스트 범위를 파악하는 출발점 |
| Jenkins | 빌드와 테스트 대상 버전이 만들어지는 흐름을 이해하는 데 도움을 준 CI 도구 |
| 빌드 | 테스트해야 할 제품 버전 또는 설치 파일 단위 |
| MCP | Codex가 Notion, Jira 같은 외부 도구의 문맥을 함께 참고하도록 연결하는 방식 |
| 스타터 키트 | 화면 경로, 설정 예시, 자동화 스크립트, 테스트 코드를 묶은 QA 작업 시작 패키지 |

## 가장 크게 배운 점

처음에는 용어가 각각 따로 보였지만, 테스트를 하면서 하나의 흐름으로 연결되었습니다.

```text
Jira 티켓에서 변경 이유를 읽고
Notion 문서에서 업무 맥락과 화면 경로를 찾고
빌드 버전을 확인한 뒤
PCFILTER 에이전트와 관리자 정책을 테스트하고
로그와 리포트로 결과를 검증한다
```

QA에서 용어를 안다는 것은 단어를 외우는 것이 아니라, 그 단어가 실제 테스트 흐름에서 어디에 쓰이는지 이해하는 일이었습니다.
