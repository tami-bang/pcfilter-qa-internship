# 업무 도구와 협업 흐름에서 배운 점

PCFILTER QA 인턴십에서는 제품 기능을 확인하는 일뿐 아니라, 팀이 어떤 방식으로 일을 기록하고 전달하는지도 함께 배웠습니다.

이 문서는 M-EP에서 접한 업무 흐름 중 외부에 공개해도 되는 수준의 학습 내용을 정리한 것입니다. 내부 페이지 URL, 실제 티켓 번호, 빌드 서버 정보, 계정 정보는 포함하지 않습니다.

## Notion: 팀 지식을 구조화하는 법

M-EP는 팀 온보딩, 가이드 문서, 업무 DB, 회의록, 제품 문서가 연결된 작업 공간이었습니다. 처음에는 필요한 문서를 찾는 것부터 어려웠지만, 점차 문서가 어떤 기준으로 묶이고 연결되는지 이해하게 되었습니다.

Notion을 사용하면서 배운 점:

- 긴 문서는 목차와 섹션을 나눠야 다시 찾기 쉬움
- 단순 경로 목록보다 업무 질문에서 출발하는 문서가 실제로 더 잘 쓰임
- 테스트 문서는 재현 조건, 기대 결과, 실제 결과, 증거가 분리되어야 읽는 사람이 빠르게 따라갈 수 있음
- 내부 문서와 공개 문서는 목적이 다르므로 공개 문서에서는 구조와 판단만 남기고 운영 정보는 제거해야 함

## Jira: 티켓을 테스트 시나리오로 바꾸는 법

Jira 티켓은 단순한 작업 목록이 아니라 변경 이유와 확인 범위를 파악하는 출발점이었습니다. 티켓을 읽으며 어떤 기능이 바뀌었는지, 어떤 조건에서 문제가 발생했는지, 어떤 결과를 기대하는지 정리했습니다.

Jira를 통해 익힌 점:

- 티켓 제목만으로 판단하지 않고 설명, 조건, 기대 결과를 함께 봐야 함
- 재현 단계가 모호하면 테스트 전에 확인 질문을 정리해야 함
- 같은 변경이라도 사용자 화면, 정책, 로그/리포트에 영향을 줄 수 있음
- QA 결과는 “됨/안 됨”보다 조건과 근거를 남겨야 다시 확인할 수 있음

## Jenkins: 빌드 흐름을 QA 관점에서 이해하기

Jenkins를 직접 운영한 것은 아니지만, 팀 문서를 통해 빌드 서버와 빌드 설정, 테스트 대상 버전이 만들어지는 흐름을 간접적으로 접했습니다.

이 경험을 통해 QA가 단순히 전달받은 설치 파일만 확인하는 것이 아니라, 어떤 빌드를 테스트하는지 이해하는 일이 중요하다는 점을 알게 되었습니다.

Jenkins 관련 학습 포인트:

- 테스트 대상 빌드가 만들어지는 과정이 QA 일정과 연결됨
- 같은 기능이라도 빌드 버전에 따라 재현 여부가 달라질 수 있음
- 빌드 흐름을 이해하면 테스트 결과를 전달할 때 버전과 조건을 더 정확히 남길 수 있음
- CI/CD 도구는 개발자만의 영역이 아니라 QA가 테스트 맥락을 이해하는 데도 도움이 됨

## 정리

인턴십을 통해 Notion, Jira, Jenkins를 각각 따로 보는 것이 아니라 하나의 흐름으로 이해하게 되었습니다.

```text
Notion에서 업무 맥락을 찾고
Jira에서 변경 이유와 확인 범위를 읽고
Jenkins 흐름을 통해 테스트 대상 빌드를 이해하고
PCFILTER에서 실제 동작을 확인한다
```

이 흐름을 경험하면서 QA는 제품을 확인하는 역할이면서 동시에 팀의 기록과 개발 흐름을 이어 읽는 역할이라는 것을 배웠습니다.

관련해서 인턴 기간 동안 익힌 주요 용어는 [8주 동안 익힌 용어 정리](terminology.md)에 따로 정리했습니다.
