# 개발자로 성장하기 위해 얻은 기준

PCFILTER QA 인턴십은 제품을 테스트한 경험이었지만, 동시에 앞으로 개발자가 되기 위해 어떤 기본기를 갖춰야 하는지 배운 시간이기도 했습니다.

QA 관점에서 제품을 보니, 기능은 코드 안에서만 끝나지 않았습니다. 사용자의 행동, 정책 설정, 예외 상황, 로그와 리포트, 기존 기능과의 관계까지 이어져 있었습니다.

## 기능은 흐름 안에서 동작해야 한다

기능 하나가 잘 동작하는지 확인하려면 버튼이나 메뉴만 보면 부족했습니다.

개발할 때 참고해야겠다고 느낀 점:

- 사용자가 어떤 순서로 기능에 접근하는지 생각하기
- 설정이 바뀌었을 때 화면, 동작, 기록이 함께 맞는지 보기
- 한 기능을 고쳤을 때 주변 기능이 깨지지 않는지 확인하기
- 성공 케이스뿐 아니라 비활성화, 권한 부족, 빈 결과 같은 상황도 고려하기

## 재현 가능한 기록이 중요하다

QA 결과를 남기면서 가장 중요했던 것은 “다시 따라 할 수 있는가”였습니다.

개발자로 일할 때도 같은 기준이 필요하다고 느꼈습니다.

- 문제가 발생한 조건을 남기기
- 기대 결과와 실제 결과를 분리하기
- 로그나 상태값처럼 확인 가능한 근거를 남기기
- 모호한 설명보다 재현 단계로 말하기

## 로그와 메시지는 사용자를 위한 인터페이스다

테스트를 하다 보면 화면 메시지나 로그가 불분명해서 원인을 파악하기 어려운 경우가 있습니다. 이 경험을 통해 로그와 안내 문구도 기능의 일부라는 것을 알게 되었습니다.

앞으로 개발할 때 참고할 점:

- 실패했을 때 사용자가 다음 행동을 알 수 있게 메시지 쓰기
- 개발자와 QA가 원인을 추적할 수 있는 로그 남기기
- 같은 상태를 화면, 로그, 리포트에서 서로 다르게 표현하지 않기
- 내부 용어가 사용자에게 그대로 노출되지 않도록 주의하기

## 자동화는 사람의 판단을 돕기 위한 것이다

테스트 가이드 자동화를 만들면서 자동화가 모든 판단을 대신할 수 없다는 것도 배웠습니다. 자동화는 반복되는 초안을 줄여주지만, 최종 판단은 사람이 해야 했습니다.

그래서 자동화를 만들 때 중요하다고 느낀 기준은 다음과 같습니다.

- 기존 사용자의 작성 내용을 덮어쓰지 않기
- 실행 전 dry-run이나 검증 단계를 두기
- 결과를 사람이 읽고 수정할 수 있게 만들기
- 실패했을 때 왜 멈췄는지 설명하기

## 협업 도구를 읽는 능력도 개발 기본기다

Notion, Jira, Jenkins를 접하면서 개발은 코드만 쓰는 일이 아니라는 것을 배웠습니다.

- Notion 문서에서 업무 맥락을 찾는 법
- Jira 티켓에서 변경 이유와 테스트 조건을 읽는 법
- Jenkins 문서를 통해 빌드와 버전 흐름을 이해하는 법
- 팀이 남긴 기록을 바탕으로 다음 작업을 이어가는 법

## 정리

이 인턴십을 통해 개발할 때 참고해야 할 기준이 조금 더 선명해졌습니다.

```text
기능은 사용자 흐름 안에서 생각하고
문제는 재현 가능하게 남기고
로그와 메시지는 읽는 사람을 위해 설계하고
자동화는 사람의 판단을 안전하게 돕도록 만든다
```

QA 경험은 개발에서 멀리 떨어진 일이 아니라, 더 좋은 기능을 만들기 위해 먼저 배워야 할 관점이었습니다.
